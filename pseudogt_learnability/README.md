# PseudoGT Learnability

This project is separated from `dynamic_quality_aware_classwise_aggregation`
so that pseudo-GT learning can be tested without the DQA notebooks and
aggregation experiments becoming the main variable.

## Goal

The core question is:

> Under the same BDD100K scene-client conditions used by DQA, can clients learn
> from pseudo-GT in a way that improves the detector before we design a smarter
> class-wise aggregation rule?

This matters because the DQA idea assumes that each client can produce a useful
specialized update. If pseudo-GT local learning is weak or harmful, class-wise
client selection will only select between noisy or nearly identical updates.

## Fixed Conditions

The standalone condition sheet is:

```text
pseudogt_learnability/EXPERIMENT_CONDITIONS.md
```

- Dataset/protocol: BDD100K scene clients from the existing FedSTO scene setup.
- Clients:
  - client 0: `highway`
  - client 1: `citystreet`
  - client 2: `residential`
- Server data: the same cloudy server train/validation lists generated by
  `setup_fedsto_scene_reproduction.py`.
- Evaluation splits: `highway`, `citystreet`, `residential`, and their union.
- Model family: EfficientTeacher YOLOv5L.
- Image size: 640.
- Starting checkpoint: copied from the DQA 08_4 warmup model:
  `dynamic_quality_aware_classwise_aggregation/efficientteacher_dqa08_4_scene_phase1_dqa_sba_full_policy/global_checkpoints/round000_warmup.pt`
- Local target data: unlabeled-only client training, matching the DQA/FedSTO
  target-list condition.
- All new results are written under `pseudogt_learnability/output`.

The copied warmup checkpoint is stored at:

```text
pseudogt_learnability/checkpoints/round000_warmup.pt
```

## Baseline From Previous DQA 08_4 Results

The 08_4 scene evaluation showed:

| checkpoint | scene_total mAP@0.5 | scene_total mAP@0.5:0.95 |
| --- | ---: | ---: |
| warmup | 0.352 | 0.189 |
| phase1 round003 | 0.375 | 0.203 |
| phase1 round012 | 0.369 | 0.199 |
| phase2 round001 | 0.370 | 0.203 |
| phase2 round012 | 0.348 | 0.189 |

Interpretation:

- Phase 1 can improve over warmup even when pseudo-GT is imperfect.
- Long Phase 2 tends to drift back down.
- Therefore the next useful question is not "which client should DQA select?"
  but "which pseudo-GT training setting creates learnable, non-destructive
  client updates?"

## 01 Experiment

Notebook:

```text
pseudogt_learnability/notebooks/01_best_pseudogt_learnability.ipynb
```

Runner:

```text
pseudogt_learnability/scripts/run_pseudogt_learnability_01.py
```

The 01 experiment starts every client from the copied warmup checkpoint and
tests three pseudo-GT learning profiles:

| profile | train scope | intent |
| --- | --- | --- |
| `backbone_obj_safe` | backbone | Preserve the head and test the Phase-1-like hypothesis that pseudo-GT mostly helps feature adaptation. |
| `neck_head_high_precision` | neck/head | Use fewer, higher-precision pseudo boxes to see whether the detector head can adapt without broad drift. |
| `all_consistency_lowlr` | all | Low-LR full-model adaptation with conservative tri-stage objectness gating. |

For each profile, the runner saves:

- per-client checkpoints
- an aggregate checkpoint
- an optional one-epoch supervised server repair checkpoint
- a manifest and checkpoint table
- optional scene-protocol evaluation outputs

The intended success criteria for pseudo-GT learnability are:

- best aggregate or server-repaired checkpoint exceeds warmup on scene-total
  mAP@0.5:0.95, or
- at least one client-specific checkpoint improves its matching scene split
  without causing a large collapse on the total split.

If neither condition holds, DQA should not be tuned further until pseudo-GT
learning itself is changed.

## 02 Experiment

Notebook:

```text
pseudogt_learnability/notebooks/02_stable_pseudogt_learnability.ipynb
```

Runner:

```text
pseudogt_learnability/scripts/run_pseudogt_learnability_02.py
```

The 02 experiment changes the pseudo-GT generation step before changing DQA.
It uses the copied warmup checkpoint as a single teacher and predicts each
client target image twice:

- original image
- horizontal-flip image, de-augmented back to original coordinates

Only boxes that keep the same class and nearly the same location across views
are written as offline YOLO pseudo labels.  Client training then uses the same
source labeled cloudy list plus the stable target pseudo-label list, so the
source data remains an anchor while the target scene can still specialize the
client.

The default 02 profiles are:

| profile | scope | intent |
| --- | --- | --- |
| `stable_mix_backbone` | backbone | Test whether stable target boxes can reproduce the Phase-1-like feature benefit. |
| `stable_mix_all_lowlr` | all | Test whether full-model adaptation becomes safe when bbox labels are stability-gated. |
| `stable_mix_neck_head` | neck/head | Optional high-precision head adaptation profile. |

Key pseudo-label diagnostics are saved to:

```text
output/02_stable_pseudogt/stats/02_pseudo_label_stats.csv
output/02_stable_pseudogt/stats/02_pseudo_label_stats.json
```

The important first check is whether stable pseudo labels are numerous enough
and not dominated by one easy class.  Only after that should the mAP comparison
against `warmup_global` be interpreted.

## Repair-Oriented Direction

After 02, the main objective is reframed from pseudo-GT-only client improvement
to multi-round repaired-global improvement.  The current plan, metrics, and
decision rules are documented in:

```text
pseudogt_learnability/REPAIR_ORIENTED_PSEUDOGT.md
```

Notebook 03 implements this direction:

```text
pseudogt_learnability/notebooks/03_repair_oriented_multiround_pseudogt.ipynb
```

Runner:

```text
pseudogt_learnability/scripts/run_pseudogt_learnability_03.py
```

03 evaluates repaired global checkpoints after every round.  Client and
aggregate checkpoints are diagnostic; the primary metrics are saved to:

```text
output/03_repair_oriented_multiround/stats/03_round_metrics.csv
output/03_repair_oriented_multiround/stats/03_round_metrics_summary.json
```
